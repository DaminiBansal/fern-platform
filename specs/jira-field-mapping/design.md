# Design: JIRA Field Mapping

## Overview

Adds a `JiraFieldMapping` aggregate to the existing `integrations` domain. A
project manager uses it to record which JIRA fields map onto Fern's eight
requirement fields, stored per Fern project. The implementation extends the
existing `integrations` package (which already owns `JiraConnection`), adds a
GORM repository and migration, exposes GraphQL queries and mutations on the
existing schema, and adds a Field Mapping section to the static frontend
matching `jira-field-mapping-ui-mock.png`. Issue #27 (sync) reads the persisted
mapping to translate JIRA issues into Fern requirements.

## UI Placement

The Field Mapping screen lives **within the Project detail page**, on the
existing **Integrations tab** (`web/index.html` line ~8496, tab id
`integrations`). It does not get its own top-level nav item.

The Integrations tab already contains `JiraConnectionManager` (~line 8657).
The Field Mapping section is added directly below it, making the full
configure → map → sync flow visible in one place:

```
Project detail  ▶  Integrations tab
├── JIRA Connection  (existing JiraConnectionManager component)
├── Field Mapping    ← this feature (rendered only when a connection exists)
└── Sync             (issue #27, rendered only when a mapping is saved)
```

The Field Mapping section is conditionally rendered: it appears only when the
project has an active JIRA connection. If no connection exists, the section
shows a short prompt ("Connect to JIRA first") with a scroll-anchor to the
connection panel above it. This matches Requirement 6 ac4 and replaces the
standalone-route approach described in Task 6.1–6.3 — there is no separate
`/projects/{id}/jira-mapping` route; the JS is loaded as part of the project
detail page and the section is toggled into view.

## Data Flow

### Loading the Field Mapping screen

1. User navigates to the project's Integrations tab; the `FieldMappingModal`
   opens when they click "Configure Mapping" on an active JIRA connection.
2. JS issues a GraphQL query: `jiraFieldMapping(projectId)` plus
   `jiraConnection(projectId)` plus `jiraFields(connectionId)`.
3. Resolver `JiraFieldMapping_domain` authorizes the caller (project
   manager or admin), then calls `JiraFieldMappingService.Get(projectId)`.
4. Service returns the saved mapping or, if none exists, the default mapping
   (Requirement ID → Issue Key, Title → Summary, …) as an unsaved snapshot.
5. Resolver `JiraFields_domain` calls `DefaultJiraClient.ListFields()` which
   issues `GET <jiraUrl>/rest/api/2/field` and returns the list of standard
   JIRA fields (custom fields are filtered out per the constraint).
6. Browser renders rows with the loaded selections and JIRA-field options.

### Saving a Field Mapping

1. User changes selections; client-side validation flags conflicts and
   missing required fields, disabling Save while invalid.
2. User clicks Save → JS issues
   `saveJiraFieldMapping(input: SaveJiraFieldMappingInput!)`.
3. Resolver authorizes, then calls `JiraFieldMappingService.Save(input)`.
4. Service validates: both required Fern fields are mapped; no JIRA field
   maps to more than one Fern field; if any single-value Fern field maps to
   a multi-value JIRA field, a reduction strategy is present.
5. On success: upsert into `jira_field_mappings` (one row per project),
   record `updated_by` and `updated_at`, emit a logrus structured log,
   return the saved snapshot.
6. On failure: return a typed validation error listing the offending fields;
   no DB write occurs.

### Effect on JIRA sync (informational — issue #27 wiring)

1. Sync runner calls `JiraFieldMappingService.Get(projectId)`.
2. If no mapping exists, sync aborts with a clear error surfaced to the
   project manager (Req 8 ac2).
3. Otherwise, sync uses the mapping (including reduction strategies) when
   translating each JIRA issue into a Fern requirement.

## Interface Specifications

### Domain types (in `internal/domains/integrations/`)

```go
// FernField identifies a Fern requirement field that can receive a JIRA
// value during sync.
type FernField string

const (
    FernFieldRequirementID          FernField = "requirement_id"
    FernFieldRequirementTitle       FernField = "requirement_title"
    FernFieldRequirementDescription FernField = "requirement_description"
    FernFieldParentRequirement      FernField = "parent_requirement"
    FernFieldRequirementType        FernField = "requirement_type"
    FernFieldReleaseVersion         FernField = "release_version"
    FernFieldRequirementStatus      FernField = "requirement_status"
    FernFieldTags                   FernField = "tags"
)

// ReductionStrategy describes how a multi-value JIRA field collapses into
// a single-value Fern field.
type ReductionStrategy string

const (
    ReductionFirst       ReductionStrategy = "first"
    ReductionConcatenate ReductionStrategy = "concatenate"
    ReductionPerValue    ReductionStrategy = "per_value"
)

// JiraFieldMapping is the aggregate persisted per Fern project.
type JiraFieldMapping struct {
    id        string
    projectID string
    entries   []FieldMappingEntry
    updatedBy string
    createdAt time.Time
    updatedAt time.Time
}

type FieldMappingEntry struct {
    FernField         FernField
    JiraFieldID       string             // empty = unmapped
    JiraFieldName     string             // cached display name
    Reduction         *ReductionStrategy // non-nil iff multi-value JIRA → single Fern
}
```

`JiraFieldMapping` enforces invariants in a constructor and in
`UpdateEntries()`:
- exactly one entry per Fern field
- required Fern fields must have non-empty `JiraFieldID`
- no two entries share the same non-empty `JiraFieldID`
- single-value Fern fields mapped to multi-value JIRA fields must carry a
  reduction strategy

### Migration: `000022_create_jira_field_mappings_table`

```sql
CREATE TABLE IF NOT EXISTS jira_field_mappings (
    id           BIGSERIAL PRIMARY KEY,
    project_id   VARCHAR(36) NOT NULL UNIQUE,
    entries      JSONB       NOT NULL,
    updated_by   VARCHAR(255) NOT NULL,
    created_at   TIMESTAMP   NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at   TIMESTAMP   NOT NULL DEFAULT CURRENT_TIMESTAMP,
    deleted_at   TIMESTAMP,
    CONSTRAINT fk_jira_field_mapping_project
        FOREIGN KEY (project_id)
        REFERENCES project_details(project_id)
        ON DELETE CASCADE
);

CREATE INDEX idx_jira_field_mapping_project_id ON jira_field_mappings (project_id);
CREATE INDEX idx_jira_field_mapping_deleted_at ON jira_field_mappings (deleted_at);
```

`entries` is JSONB rather than a join table because (a) the set of Fern
fields is small and fixed (≤8), (b) entries are always read and written as
a single atomic unit, and (c) JSONB matches the GORM repository pattern
already used in adjacent tables.

## Technical Decisions

### 1. JSONB column for entries (not a join table)

- **Choice:** Store the eight (or fewer) `FieldMappingEntry` rows in a
  single `entries JSONB` column on `jira_field_mappings`.
- **Rationale:** Entries are always read and written together; concurrent
  per-row edits are not a use case; this keeps the migration minimal and
  reads to one query. JSONB is supported natively by Postgres and is used
  elsewhere in the codebase.
- **Alternatives considered:** A `jira_field_mapping_entries` join table
  with one row per (mapping, fern_field). Rejected because it adds a
  second migration, an additional GORM model, and N+1 read risk for what
  is conceptually one record.

### 2. One mapping row per Fern project (UNIQUE on project_id)

- **Choice:** Enforce a single mapping per project via a unique constraint
  on `project_id`; `Upsert` is the only write operation.
- **Rationale:** Requirement 5 ac3 (mappings are independent per project)
  is naturally enforced; "last writer wins" with row-level locking
  satisfies Req 9 ac4 without needing optimistic versioning.
- **Alternatives considered:** Versioned mappings keyed by `(project_id,
  version)` with history retention. Out of scope; auditing is satisfied
  by `updated_by` + logrus + the existing access log (Req 7 ac3).

### 3. Reduction strategy stored alongside the mapping entry

- **Choice:** Each entry carries an optional `reduction_strategy` that is
  set only when the entry maps a single-value Fern field to a multi-value
  JIRA field.
- **Rationale:** Sync needs the strategy in one fetch with the mapping;
  storing it on the entry keeps the schema simple and the validation
  local to the aggregate.
- **Alternatives considered:** A separate `reductions` table or a global
  default strategy. Both add coupling without removing the per-mapping
  choice the user must make.

### 4. GraphQL only — no REST surface

- **Choice:** Expose the feature through GraphQL queries and mutations on
  the existing schema; no new REST routes.
- **Rationale:** The web UI is the only client; GraphQL is already the
  primary API for browser code (Req 9 ac3); the test-run REST endpoint is
  reserved for client libraries submitting results.
- **Alternatives considered:** REST endpoints under `/api/v2/projects/
  {id}/jira-field-mapping`. Rejected — duplicate surface for no clear
  consumer.

### 5. Filter out custom JIRA fields in `ListFields`

- **Choice:** `DefaultJiraClient.ListFields` includes only fields where
  `Custom == false`.
- **Rationale:** Custom fields are out of scope for this iteration
  (Constraints). Filtering server-side keeps the selectable set small,
  avoids confusing the user, and prevents a future user from saving
  a mapping that the sync feature does not yet understand.
- **Alternatives considered:** Include custom fields and warn in the UI.
  Rejected — encourages mappings we cannot yet support.

### 6. Defaults computed in the service, not stored

- **Choice:** When a project has no saved mapping, the service returns a
  default snapshot (not persisted) for the UI to render.
- **Rationale:** Avoids a "create defaults on connection" side effect at
  connection time; keeps the persistence model simple (a row exists iff
  the user has saved).
- **Alternatives considered:** Auto-persist defaults when a JIRA
  connection is created. Rejected — surprising behaviour and creates a
  cleanup obligation when the connection is removed.

### 7. Frontend already delivered as inline React in `index.html` (issue #25)

- **Choice:** Issue #25 implemented `FieldMappingModal` as an inline React
  component in `web/index.html` (line ~7638) rather than a separate
  `web/js/jira-field-mapping.js` module. This feature extends that component
  in place rather than extracting it.
- **What remains:** Replace the hardcoded JIRA field list and hardcoded default
  mappings with real GraphQL calls; wire the commented-out `SAVE_FIELD_MAPPINGS`
  mutation; align Fern field IDs with domain constants (`requirement_title`,
  `requirement_type`, `release_version`, `requirement_status`); remove the two
  extra fields (`priority`, `component`) that are not in the spec's eight-field
  set.
- **Alternatives considered:** Extract to `web/js/jira-field-mapping.js`.
  Rejected — unnecessary churn given the component already exists and works.

## Error Handling

| Scenario | Detection | Response |
|---------|-----------|----------|
| Required Fern field unmapped on save | `JiraFieldMapping` constructor / `UpdateEntries` validation | Return `ErrRequiredFieldUnmapped` listing missing fields; GraphQL surfaces it as a typed validation error; no DB write |
| Same JIRA field mapped twice | Aggregate validation in service | Return `ErrDuplicateJiraField` listing the duplicate field id and the conflicting Fern fields; no DB write |
| Multi-value JIRA → single-value Fern without reduction | Aggregate validation | Return `ErrMissingReductionStrategy(fernField)`; no DB write |
| Unauthorized user attempts read or write | Resolver-layer authorization (same pattern as `domain_resolvers.go` post-PR #156) | Return GraphQL "access denied" error; log to access log; no service call |
| No JIRA connection configured for project | Service checks via `JiraConnectionRepository.FindByProjectID` | Return `ErrNoJiraConnection`; UI hides selects and shows the connection prompt (Req 6 ac4). Note: an untested (inactive) connection is sufficient to save a mapping — liveness is only required at sync time (issue #27). |
| JIRA `/rest/api/2/field` request fails | `DefaultJiraClient.ListFields` HTTP error path | Surface as GraphQL error on `jiraFields`; UI shows a banner and disables Save; logrus error log |
| DB write fails (Postgres down, FK violation) | GORM error returned from `Upsert` | Wrap with context, return GraphQL error; UI shows transient error toast; logrus error log |
| Concurrent save lost-update risk | Row-level lock via `SELECT ... FOR UPDATE` inside the upsert transaction | Last writer wins; the second writer's read inside the transaction observes the first writer's state |
| Sync started without a saved mapping (Req 8 ac2) | Sync code path (issue #27) calls `Get` and finds no row | Sync aborts with `ErrNoFieldMapping`; UI surfaces a clear "configure mapping first" message |

## Testing Approach

### Unit (Ginkgo + Gomega; alongside existing `*_test.go` files in `integrations/`)

- `JiraFieldMapping` aggregate: constructor accepts valid entries; rejects
  duplicate JIRA field IDs; rejects missing required fields; rejects
  missing reduction strategy when required; round-trips through
  `UpdateEntries`.
- `JiraFieldMappingService`:
  - `Get` returns saved snapshot when row exists; returns default snapshot
    when row is absent (table-driven test for the default set).
  - `Save` upserts on success; surfaces typed validation errors on
    invalid input; records `updated_by` from context.
  - `Save` is rejected when no JIRA connection is configured for the
    project (`FindByProjectID` returns empty); an untested/inactive
    connection is acceptable — liveness is enforced at sync time.
- `DefaultJiraClient.ListFields`: with `go-sqlmock`-style HTTP recorder /
  `httptest.NewServer`, asserts the JIRA API call shape, filters custom
  fields, and sorts results.

### Repository / database (GORM + `go-sqlmock`)

- `GormJiraFieldMappingRepository.Upsert` round-trips a mapping
  (insert path + update path).
- Soft delete via `deleted_at` is respected on `Get`.
- Foreign-key cascade: deleting a project cascades the mapping row.

### GraphQL resolver tests

- Schema generation succeeds after `go generate`.
- `jiraFieldMapping` returns default snapshot for projects without a saved
  mapping.
- `saveJiraFieldMapping` mutation returns the saved snapshot and persists
  via the service.
- Authorization: a non-manager / non-admin caller receives the standard
  access-denied error and the service is not invoked
  (follows the post-PR #156 pattern in `domain_resolvers.go`).

### Acceptance (Ginkgo, under `acceptance/`)

- Builds on the existing `acceptance/helpers/mock_jira_server.go` to
  expose a `/rest/api/2/field` endpoint returning a fixture of standard
  fields.
- End-to-end: configure a JIRA connection for a project; load the Field
  Mapping screen; save with defaults; reload and verify persistence;
  attempt to save with a conflict and verify the failure.
- Acceptance under `acceptance/jira_field_mapping_test.go`
  hits the deployed stack via GraphQL.

### Performance / smoke

- Performance is not a primary risk (≤8 rows per project, single upsert),
  but the existing performance test harness under `acceptance/
  performance/` will exercise `jiraFieldMapping(projectId)` to confirm
  the resolver returns within the existing GraphQL latency budget.
